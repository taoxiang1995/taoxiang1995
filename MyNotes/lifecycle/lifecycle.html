React Life Cycle

Q
What is the life cycle of a react component?
A
Birth / Mounting
=>props and state are defined and configured.
=>Component and all its children are mounted on to Native UI Stack (DOM)
Growth / Update
=>get new props, change state
=>handle user interaction
=>will be repeated again and again. 
Death / Unmount
=>relies component for garbage collection
=>happens when user navigate away, UI page changes, a component is hidden.


Q
What functions will be called in each of these cycles?
A
Birth / Mounting
getDefaultProps()   (in ES6: MyComponent.defaultProps)
getInitialState()  (in ES6: this.state=… in constructor)
componentWillMount( )
render( )
componentDidMount( )
Growth / Update
componentWillReceiveProps()
shouldComponentUpdate( )
render()
componentWillUpdate()
Death / Unmount
componentWillUnmount( )



Q
How to give a default props to a component?
A
import React from 'react';

Person.defaultProps = { age: 'unknown' };

Q
what is the difference between null and undefined props?
A
=> null is a defined value while defined, while, as its name suggested, is undefined. 
=>when rendering, null will be render as '' while undefined will be rendered as unknown

Q
Why do we need componentWIllMount( ) method, and how can we use it?
A
=>componentWillMount will be called after props and state are set, but before render( ) function
=>We don’t have access to Native UI and children, because they are not created yet.
What can be done in componentWillMount( )?
(1)handle configuration
(2)update state (do some calculation based on props, api request.etc.)
(3)respond to global native UI events.

Q
What can’t we do in render() function?
A
=>We can not call setState( ), because it will cause infinite loop
=>we can not query native UI DOM

Q
What is the calling order of componentDidMount if we are trying to render a complex component which includes child component?
A
componentDidMount ( ) will be called from bottom up (from child to parent)

Q
Why do we need to use componentDidMount ( )?
A
=>componentDidMount function will be called after render function, meaning we can reference Native UI dom in componentDidMount( )
Common Usage of componentDidMount ( ):
(1)access Native UI and adjust component’s current state.
(2)setting up 3rd party UIs
(3)attach generic event listener:
   window,addEventListner('resize', this.handleResize);
Ex:
var Box = React.createClass({ getInitialState: function() { return {windowWidth: window.innerWidth}; }, handleResize: function(e) { this.setState({windowWidth: window.innerWidth}); }, componentDidMount: function() { window.addEventListener('resize', this.handleResize); }, componentWillUnmount: function() { window.removeEventListener('resize', this.handleResize); }, render: function() { return <div>Current window width: {this.state.windowWidth}</div>; }}); ReactDOM.render(<Box />, mountNode);
why in componentDidMount( )? we need to bind our third party UI into the current DOM, so we need to reference Native UI
componenDidmount is the place we can reference to NativeUI
Ex:
import React from 'react';
import ReactDOM from 'react-dom';
import c3 from 'c3';

export default class Chart extends React.Component {

  componentDidMount() {
    this.chart = c3.generate({
      bindto: ReactDOM.findDOMNode(this.refs.chart),
      data: {
        columns: [
          ['data1', 30, 200, 100, 400, 150, 250],
          ['data2', 50, 20, 10, 40, 15, 25]
        ]
      }
    });
  }

  render() {
    return (
      <div ref="chart"></div>
    );
  }
}

Q
When will Growth/Update phase be triggered?
A
=>It will be triggered by three ways:
(1) update props
(2) update state
(3) calling forceUpdate( )

Q
How can we update a props?
A
=>component itself can not change its props. props have to be changed externally by its parent.
=>Ex:
parent component has a event listener which will change its state according to user interaction.
child component has a props passed in by parent which depends on parent’s state.
So whenever parent’s state changed via user interaction, the child component’s props will be changed. 

Q
Why do we need componentWillReceiveProps?
A
=>componentWillReceiveProps(nextProps)
will give us access to new props value and current props value.
We access new props value via passed in argument of componentWillReceiveProps
We access current props simply as this.props
=>We can access new props and update internal state in this method.

Q
Why do we need shouldComponentUpdate( )?
A
=>shouldComponentUpdate( ) gives us a chance to compare current state and props with next state and props, and make a decision whether to render.
=>shouldComponentUpdate will either return true or return false.
=>if return true, it will trigger a render (), if return false, it will not trigger a render. 

Q
What is the problem with React’s default pureRenderMixin ( )?
A
pureRenderMixin ( ) will do a shallow comparison, which will not properly check against mutable data. 


Q
Why do we need to use componentDidUpdate( )?
A
=>componentDidUpdate is called after all the children is updated. 
=>componentDidUpdate is called from buttom up (from child to parent)
(1) We use componentDidUpdate to manage 3rd party UI elements and interacting with Native UI. 
update the 3rd party UI with new data.
componentDidUpdate(prevProps, prevState) {
  // only update chart of the data has changed
  if (prevProps.data !== this.props.data) {
    this.chart = c3.load({
      data: this.props.data
    });
  }
}

Q
Why do we need componentWillUnmount( ) and what is the calling order of it?
A
=>componentWillUnmount( ) will be called before entering garbage collection.
=>we deleting a component, its coponentWillUnmount() will be called, and its children’s componentWillUnmount( ) will be called. 

(1) we will unregister any global/system/library events.
(2)destroy 3rd party UI element. 
(3)Remove event listener.
